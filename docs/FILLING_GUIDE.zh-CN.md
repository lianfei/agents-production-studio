# 填写手册（中文）

这份手册回答两个问题：

1. 每一步应该填什么
2. 什么样的输入更容易得到高质量结果

## 一、先理解这个系统在收集什么

系统真正要收集的是四类信息：

- 你要做什么
- 在什么环境下做
- 有哪些不能碰的边界
- 最终怎样算做成

如果你把这四类信息写清楚，最终 AGENTS.md 的质量通常就会明显更高。

## 二、逐步填写说明

## Step 1：模板与目标

### 1. 模板

作用：帮助系统建立一个初始工作框架。

建议：

- 有浏览器页面、HTTP 服务、结果页时，选 `HTTP 服务 / Web 应用`
- 主要是命令行工具时，选 `CLI / 工具链`
- 主要是多步骤批处理时，选 `自动化工作流`
- 以数据清洗、分析、标注为主时，选 `数据分析 / 处理`
- 有大量外部接口时，选 `API / 集成服务`

### 2. 行业

作用：帮助系统理解上下文语言和典型约束。

建议：

- 填“任务所服务的业务领域”，不是技术栈
- 例如：文旅、教育、零售、医疗、金融、制造

### 3. 任务描述

这是最重要的字段。

建议至少写清楚：

- 最终产物是什么
- 主要用户是谁
- 使用方式是什么
- 成功标准的大方向是什么

好的写法：

- “为一家城市文化中心搭建一个浏览器可访问的活动内容整理工作台，通过表单收集活动信息和发布要求，先生成 PLAN.md，再生成最终 AGENTS.md，并支持结果页查看与复制。”

不好的写法：

- “做一个页面”
- “帮我生成 AGENTS”

## Step 2：场景与环境

### 1. 目标用户

填谁会真正使用这个结果。

例如：

- 内部团队
- 普通业务用户
- 运维人员
- 分析师

### 2. 输出语言

决定最终 AGENTS.md 和页面内容以什么语言为主。

建议：

- 只服务中文用户时选中文
- 需要同时交付中英文内容时选双语

### 3. 运行环境

这里允许组合，不要只选一个。

常见组合：

- 浏览器 + HTTP 服务
- 本地 + 远程
- 浏览器 + Docker
- 本地 + 远程 + Docker

填法原则：

- 选“最终实际运行会涉及到的环境”
- 没有现成选项时，再补自定义环境

### 4. 风险容忍度

这是系统判断“更保守”还是“更激进”的依据。

建议：

- 低：生产任务、不能乱改、不能跳过验证
- 中：一般项目，允许合理取舍
- 高：探索性原型

## Step 3：偏好与确认

### 1. 偏好技术栈

这里填“最好采用什么”，而不是“只能采用什么”。

适合填写：

- Python
- FastAPI
- React
- Docker
- PostgreSQL

不适合填写：

- “必须保密”
- “不能影响线上”

这些应该放到“约束条件”。

### 2. 约束条件

这里填不能违背的规则。

适合填写：

- 不得跳过验证
- 记录所有关键过程并带时间戳
- 供应商必须可替换
- 必须兼容现有 API
- 必须支持 Docker 部署

### 3. 自由发挥 / 补充说明

这里适合放那些很重要、但不属于固定字段的信息。

例如：

- 页面风格偏高端官网，不要后台管理感
- 不要展示内部目录、原始文件数量和本地路径
- 结果页需要更强调业务语言，而不是工程术语

## Step 4：补充任务问答

这一阶段不是让你“从头再写一遍”，而是补充前面还缺的关键信息。

填写原则：

- 能明确回答就明确回答
- 现在还不确定的可以先忽略
- 如果问题不在选项里，优先用手动输入补充

重点字段通常包括：

- 验收标准
- 运行边界
- 远程访问方式
- Docker / 部署要求
- 输出路径或目录规则

## Step 5：PLAN 预览

这一步建议你先看 PLAN.md，再决定是否生成最终 AGENTS.md。

重点检查：

- 任务目标有没有偏
- 运行环境有没有写错
- 约束是否遗漏
- 验收标准是否太空

如果 PLAN.md 看起来已经偏了，不要急着点生成，先回前面补信息。

## 三、什么样的输入质量更高

高质量输入通常有这些特征：

- 任务描述里明确写出“最终产物”
- 环境用组合方式写清楚
- 约束是真约束，不是口号
- 验收标准可检查，而不是“做得好看”
- 自由发挥里补充风格和业务表达要求

## 四、常见错误

### 错误 1：任务描述太短

例如：

- “做一个后台”
- “优化页面”

问题：

- 系统不知道到底要交付什么
- PLAN.md 和 AGENTS.md 会变得非常泛

### 错误 2：把偏好和约束混在一起

例如把“最好用 Python”写进约束。

更好的拆法：

- 偏好技术栈：Python、FastAPI
- 约束条件：不得跳过验证、供应商必须可替换

### 错误 3：环境只写一个词

例如只写 `Docker`。

更好的写法：

- 浏览器 + HTTP 服务 + Docker
- 本地 + 远程 + Docker

### 错误 4：验收标准只写“完成即可”

验收标准应该能被检查。

## 五、好的样例

场景：城市艺术节活动协同台

```json
{
  "template_type": "http_service",
  "industry": "文化活动 / 文旅",
  "task_description": "为一家城市艺术节运营团队生成一份可执行 AGENTS.md。最终产物应包含一个浏览器可访问的工作台，通过表单收集活动主题、日程、合作方要求和上线边界，先生成 PLAN.md，再生成最终 AGENTS.md，并支持结果页查看、一键复制和下载。",
  "target_user": "internal_team",
  "output_language": "zh",
  "environment": ["browser_service", "docker"],
  "risk_tolerance": "medium",
  "preferred_stack": ["python", "fastapi", "docker"],
  "constraints": [
    "记录所有关键过程并带时间戳",
    "不得跳过验证",
    "供应商必须可替换"
  ],
  "creative_notes": "页面风格更接近主流品牌官网，不要后台管理台感，也不要暴露内部目录或本地路径。"
}
```

为什么这个样例是“好的”：

- 最终产物明确
- 用户明确
- 环境明确
- 约束明确
- 风格要求明确

## 六、非常好的样例

场景：精品酒店集团双语预订支持台升级项目

```json
{
  "template_type": "http_service",
  "industry": "酒店 / 高端服务业",
  "task_description": "为一家精品酒店集团的双语预订支持台升级项目生成一份高可执行度 AGENTS.md。最终产物需要包含浏览器可访问的向导式服务，通过步骤化界面收集项目目标、用户场景、运行环境、接口约束、验收标准与部署边界；系统需要先生成 PLAN.md，再基于 PLAN.md 生成最终 AGENTS.md；结果页必须支持查看、复制和下载，并给出适合生产上线的部署建议。",
  "target_user": "internal_team",
  "output_language": "bilingual",
  "environment": ["browser_service", "local_plus_remote", "docker"],
  "risk_tolerance": "low",
  "preferred_stack": ["python", "fastapi", "docker", "postgresql", "redis"],
  "constraints": [
    "记录所有关键过程并带时间戳",
    "不得跳过验证",
    "供应商必须可替换",
    "必须兼容现有 HTTP API",
    "必须支持 Docker 部署",
    "页面中不得暴露本地路径、原始数据目录或内部统计信息"
  ],
  "creative_notes": "整体视觉偏高端酒店官网：首页与结果页都要更重视业务语言和信任感表达，不要做成开发者后台。"
}
```

为什么这个样例是“非常好的”：

- 任务目标非常清晰
- 明确区分了功能、部署、结果页、生成顺序
- 环境是组合式填写
- 约束足够具体且可验证
- 风格和信息暴露边界都写明了

## 七、推荐做法

如果你想稳定拿到更好的结果，建议这样填写：

1. 先把任务描述写完整
2. 再补环境组合
3. 再补真实约束
4. 最后用补充问答把缺失信息补齐
5. 先看 PLAN.md，再生成最终结果
